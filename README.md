# Secure Auth App

Web: https://auth-app-lyart-omega.vercel.app

Next.js / TypeScript で作成した、セッションベース認証のWebアプリです。サインアップ、ログイン、セッション管理、パスワード変更、管理者向けユーザー管理、監査ログまでをひとつの小さなアプリとしてまとめています。

## Overview

ユーザーはメールアドレスとパスワードでアカウントを作成し、ログイン後に自分のセッションやセキュリティイベントを確認できます。管理者はユーザー一覧、アカウント停止、ロック解除、監査ログ確認を行えます。

認証はサーバー側で検証し、セッションはHttpOnly CookieとDB上のトークンハッシュで管理しています。ローカルではSQLite、VercelではPostgreSQLを使う構成です。

## Screenshot Walkthrough

### 1. サインアップ: パスワード強度と表示切り替え

![Sign up](./public/screenshots/signup-security.png)

この画面では、パスワード入力中に強度メーターがリアルタイムで更新されます。文字数、大文字、小文字、数字、記号を個別に判定し、不足している条件を改善点として表示します。パスワード欄には`Show` / `Hide`ボタンを用意し、入力内容を確認しながら安全に登録できます。

サーバー側でも同じ強度判定を行うため、画面上のチェックを回避して送信されても弱いパスワードは登録できません。登録時のパスワードは`bcrypt`でハッシュ化して保存します。

### 2. ログイン: 試行回数制限とRemember me

![Login lockout](./public/screenshots/login-lockout.png)

この画面では、ログイン失敗が短時間に続いた場合の一時ロックを確認できます。メールアドレス単位とIPアドレス単位で失敗回数を記録し、10分以内に5回失敗すると約5分間ログインを制限します。存在しないメールアドレスと誤ったパスワードのどちらでも、同じエラーメッセージを返します。

`Remember me`をオンにすると30日間のセッション、オフの場合は2時間の通常セッションになります。どちらもHttpOnly CookieとDB上のセッショントークンハッシュで管理します。

### 3. ダッシュボード: ログイン中ユーザーとセッション種別

![Dashboard](./public/screenshots/dashboard-remember-me.png)

ログイン後のダッシュボードでは、現在のユーザー情報、ロール、セッション種別を確認できます。画面に表示するユーザー情報は公開用の型に変換しており、パスワードハッシュやセッショントークンは表示しません。

管理者の場合だけ管理者向けリンクが表示されますが、実際の認可はUIだけではなくサーバー側の`role`チェックで行います。

### 4. セッション管理: アクティブセッションと個別ログアウト

![Sessions](./public/screenshots/sessions.png)

セッション管理画面では、自分のアクティブセッションを一覧できます。作成日時、最終利用日時、通常セッションかRemember meセッションか、現在のセッションかどうかを確認できます。

各セッションは個別に失効できます。失効処理はDB上の`revokedAt`を更新して行うため、対象セッションのCookieが残っていても次回アクセス時には認証されません。

### 5. セキュリティ設定: パスワード変更と監査イベント

![Security settings](./public/screenshots/security-settings.png)

セキュリティ設定画面では、現在のパスワードを確認したうえでパスワードを変更できます。新しいパスワードにも強度チェックを適用し、現在と同じパスワードの再利用を防ぎます。

パスワード変更に成功すると、現在のセッション以外をまとめて失効します。画面下部ではログイン、ログアウト、パスワード変更、セッション失効など、自分に関係するセキュリティイベントを確認できます。

### 6. 管理者ユーザー管理: 停止とロック解除

![Admin users](./public/screenshots/admin-users.png)

管理者ユーザー管理画面では、ユーザー一覧、ロール、ステータス、ログイン失敗回数、ロック状態を確認できます。管理者はユーザーのアカウント停止、再有効化、ログインロック解除を実行できます。

これらの操作はServer Action内で管理者ロールを再確認してから実行します。アカウント停止時は対象ユーザーのアクティブセッションもすべて失効します。

### 7. 管理者監査ログ: 全体の認証イベント確認

![Admin audit](./public/screenshots/admin-audit.png)

管理者監査ログでは、サインアップ、ログイン成功、ログイン失敗、ログアウト、セッション失効、パスワード変更、管理者操作、CSRFブロックなどを横断的に確認できます。イベントには対象ユーザー、操作ユーザー、IPアドレス、User-Agent、作成日時を保存します。

内部エラー、秘密情報、セッショントークン、パスワードハッシュは監査ログにも画面にも表示しません。

## Authentication Design

このアプリはセッションベース認証です。

- ログイン時にランダムなセッショントークンを生成します。
- Cookieには生のセッショントークンを保存します。
- DBにはセッショントークンのSHA-256ハッシュだけを保存します。
- Cookieは`HttpOnly`、`SameSite=Lax`、`path=/`、有効期限付きで設定します。
- Vercel上ではCookieに`Secure`が付きます。
- 通常セッションは2時間、Remember meセッションは30日有効です。
- セッションはサーバー側で毎回検証し、失効済み・期限切れ・停止ユーザーのセッションは拒否します。

## Security Architecture

### 1. パスワード保護

パスワードは平文保存せず、サインアップ時とパスワード変更時に`bcrypt`のcost 12でハッシュ化してからDBへ保存します。DBの`User`には`passwordHash`だけを保持し、画面表示やセッション情報で返すユーザー型からはパスワード関連の値を除外しています。

サインアップ画面では、入力中のパスワードをリアルタイムに判定します。チェック項目は文字数、大文字、小文字、数字、記号です。弱い場合は不足している条件を表示し、サーバー側でも同じ判定ロジックを使って、5項目中4項目以上かつ128文字以内のパスワードだけを受け付けます。クライアント側の表示だけに頼らないため、フォームを書き換えて送信されても弱いパスワードは登録できません。

確認方法:

- `/signup`で短いパスワードや記号なしのパスワードを入力すると、強度と改善点が表示されます。
- 許容基準を満たさないまま送信すると、サーバー側でも登録が拒否されます。
- `/security`のパスワード変更でも、現在のパスワード確認、強度チェック、同じパスワードの再利用防止を行います。

### 2. パスワード表示切り替え

ログイン、サインアップ、パスワード変更の各フォームで、パスワード欄の表示 / 非表示を切り替えられます。切り替えは`type="password"`と`type="text"`の変更だけで行い、入力値を別の場所へ保存したり、ログへ出力したりしません。

確認方法:

- `/login`または`/signup`でパスワード欄の`Show` / `Hide`ボタンを押すと、同じ入力欄のまま表示状態だけが切り替わります。

### 3. セッションベース認証

認証方式はセッションベースです。ログイン時に`crypto.randomBytes(32)`で推測困難なランダムトークンを生成し、生のトークンはCookieにだけ保存します。DBにはトークンそのものではなく、SHA-256ハッシュを保存します。これにより、DB内のセッションレコードだけを見てもCookieとして使える生トークンは得られません。

リクエストごとにサーバー側でセッションを検証し、次の状態ではログイン済みとして扱いません。

- DBに一致するセッションがない
- セッションが期限切れ
- セッションが個別ログアウト済み
- ユーザーが停止状態

有効なセッションでは`lastUsedAt`を更新し、アクティブセッション一覧で作成日時、最終利用日時、現在のセッションかどうかを確認できます。

確認方法:

- ログイン後に`/sessions`を開くと、現在のセッションと他の有効セッションが表示されます。
- 別ブラウザやシークレットウィンドウでログインすると、複数セッションとして確認できます。
- 任意のセッションを`Sign out`すると、そのセッションだけが無効化されます。

### 4. CookieとRemember me

セッショントークンはCookieで管理し、`HttpOnly`、`SameSite=Lax`、`path=/`、有効期限を設定しています。Vercelなど本番環境では`Secure`も付くため、HTTPS通信でのみCookieが送信されます。トークンは`localStorage`やクライアント側の状態には保存しません。

Remember meを使わない通常ログインは2時間、Remember meを有効にしたログインは30日間のセッションとして保存します。ログアウト時はDB上のセッションを失効させ、Cookieも削除します。

確認方法:

- `/login`で`Remember me`をオンにしてログインすると、`/dashboard`や`/sessions`でRemember meセッションとして表示されます。
- オフの場合は通常セッションとして表示されます。

### 5. ログイン試行回数制限

ログイン失敗はメールアドレス単位とIPアドレス単位の両方で記録します。10分以内に5回失敗すると、約5分間ログインを制限します。存在しないメールアドレスと誤ったパスワードのどちらでも、画面には同じエラーメッセージを返すため、メールアドレスの存在確認に使われにくい設計です。

ログインに成功すると、そのユーザーとアクセス元に紐づく失敗記録をクリアします。管理者はユーザー管理画面からロック解除もできます。

確認方法:

- `/login`で同じメールアドレスに対して誤ったパスワードを5回入力すると、一時ロックのメッセージが表示されます。
- 管理者で`/admin/users`を開くと、対象ユーザーのロック解除を実行できます。

### 6. サーバー側の認可

ログイン必須画面はサーバー側でセッションを確認します。管理者画面と管理者操作は`role`が`ADMIN`のユーザーだけを許可し、一般ユーザーがURLを直接開いてもダッシュボードへ戻されます。管理者用のユーザー停止、再有効化、ロック解除もServer Action内で管理者チェックを通してから実行します。

アカウント停止時は対象ユーザーの有効セッションをすべて失効させます。停止されたユーザーは既存Cookieを持っていても次回アクセス時に認証が通らず、ログインも拒否されます。

確認方法:

- 一般ユーザーで`/admin/users`や`/admin/audit`へアクセスすると、管理者画面は表示されません。
- 管理者でユーザーを停止すると、対象ユーザーのセッションが無効化されます。

### 7. CSRF対策

状態を変更するServer Actionでは、取得できる`Origin`ヘッダーと`Host` / `x-forwarded-host`を比較します。一致しない場合は処理を止め、`csrf_blocked`としてセキュリティイベントに記録します。Cookieの`SameSite=Lax`と組み合わせて、外部サイトからの意図しないフォーム送信を通しにくくしています。

確認方法:

- 通常の画面操作ではそのまま処理されます。
- 異なるOriginから同じServer Actionへ送信された場合は拒否され、監査ログにブロックイベントが残ります。

### 8. セキュリティイベントと監査ログ

サインアップ、ログイン成功、ログイン失敗、ログアウト、セッション失効、他セッション一括ログアウト、パスワード変更、管理者操作、CSRFブロックをDBへ記録します。一般ユーザーは`/security`で自分に関係するイベントを確認でき、管理者は`/admin/audit`で全体の監査ログを確認できます。

記録する`summary`、IPアドレス、User-Agentは長さを制限して保存しています。画面には内部エラー、秘密情報、セッショントークン、パスワードハッシュを表示しません。

確認方法:

- ログイン、ログアウト、パスワード変更、セッション失効を実行したあと`/security`を開くとイベントが増えます。
- 管理者で`/admin/audit`を開くと、ユーザー管理操作を含むイベントを確認できます。

### 9. セキュリティヘッダー

全ルートにセキュリティヘッダーを設定しています。

- `Content-Security-Policy`: 読み込み元を基本的に同一オリジンへ制限し、`frame-ancestors 'none'`で埋め込みを拒否します。
- `Strict-Transport-Security`: 本番HTTPS利用を前提に、ブラウザへHTTPS接続を促します。
- `X-Frame-Options: DENY`: クリックジャッキング対策としてフレーム表示を拒否します。
- `X-Content-Type-Options: nosniff`: MIMEタイプ推測を抑止します。
- `Referrer-Policy: strict-origin-when-cross-origin`: 外部サイトへ送る参照元情報を抑えます。
- `Permissions-Policy`: カメラ、マイク、位置情報、決済APIを無効化します。

Next.jsの動作に必要な範囲で、CSPには`unsafe-inline` / `unsafe-eval`を許可しています。より厳密な運用にする場合はnonceベースCSPへ移行できます。

確認方法:

- ブラウザのDevToolsや`curl -I`でレスポンスヘッダーを確認できます。

### 10. 入力検証と秘密情報管理

ログイン、サインアップ、パスワード変更では`zod`でメール形式、文字数、必須項目をサーバー側検証します。メールアドレスは小文字に正規化し、DBのユニーク制約で重複登録を防ぎます。

DB接続文字列などの秘密情報は環境変数から読み込みます。`.env`や`.env.local`はGitへ含めず、ローカルとVercelで別々のDBを使います。

## Demo Accounts

ローカルで`npm run local:setup`を実行すると、次のデモアカウントが作成されます。

```text
User:
email: user@example.com
password: Password123!

Admin:
email: admin@example.com
password: AdminPassword123!
```

Vercel側のデモアカウントは、Vercel上のPostgreSQLに対して作成したユーザーを管理者化して使います。環境ごとにDBが分かれるため、ローカルのデモアカウントはVercel上には自動反映されません。

## Local Development

```bash
npm install
copy .env.example .env
npm run local:setup
npm run build
npm run start
```

ローカルURL:

```text
http://localhost:3000
```

`npm run local:setup` はSQLite DBを初期化し、Prisma ClientをSQLite用に生成してからデモアカウントを作成します。

## Vercel Deployment

VercelではPostgreSQLを使用します。Vercel StorageやNeon、SupabaseなどでPostgreSQLを作成し、Project SettingsのEnvironment Variablesに次を設定します。

```text
DATABASE_URL=postgresql://...
```

`vercel.json`でBuild Commandを固定しています。

```bash
npm run vercel-build
```

実行内容:

```bash
prisma generate && prisma migrate deploy && next build
```

これにより、デプロイ時にPrisma Client生成、migration適用、Next.js buildが実行されます。
